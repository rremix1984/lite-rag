# LiteRAG v2 路线图（对齐 AnythingLLM 能力的渐进式升级）

> 目标：在保持 LiteRAG “可离线部署、可控、易维护”的基础上，分阶段升级为更强的企业级知识库与智能问答平台。  
> 原则：**小步快跑、每阶段可交付、可回滚、可验证**，避免一次性大改导致系统不可用。

---

## 0. 当前基线（进入 v2 前）

### 当前已有能力（v1）
- 登录认证（JWT）与管理员配置页。
- 知识库（workspace）CRUD。
- 文档上传解析（PDF/Word/Excel/TXT/MD/CSV）。
- 分块、向量化、pgvector 检索、SSE 流式问答。
- 引用来源展示、基础系统设置。

### 当前主要短板（v2 升级动机）
- 模型/Embedding 生态有限，Provider 兼容性不稳定。
- 检索策略较单一（纯向量 TopN + 阈值）。
- 任务处理是同步链路，缺少后台队列与重试机制。
- 可观测性、权限细粒度、审计、安全能力较弱。
- 运维自动化与回归测试体系不完整。

---

## 1. v2 设计原则

- **阶段闭环**：每个阶段都必须有清晰交付物和验收标准。
- **兼容优先**：保持现有 API 与前端主流程可用，新增能力尽量向后兼容。
- **可回滚**：所有数据结构变更必须提供迁移与回滚脚本。
- **可观测**：新增功能默认要有日志、错误码、运行状态可见性。
- **生产可用**：每阶段都需要基础压测与异常场景验证。

## 3. 分阶段详细规划

## Phase 1（M1）：数据与检索稳定性工程

### 目标
- 解决“上传成功但检索不到”“维度漂移”“文件名乱码”“脏数据残留”等稳定性问题。

### 工作项
- 数据层治理
  - 增加 `documents` 与 `vectors` 一致性巡检任务（定时）。
  - 引入孤儿数据自动修复策略（可配置：仅告警 / 自动清理）。
  - 增加向量维度检查 API：返回数据库维度、当前模型实际维度、配置维度。
- 上传链路加固
  - 统一事务/补偿逻辑：插入失败时保证 documents/vectors 一致回滚。
  - 文件名编码标准化（UTF-8 正规化 + 历史兼容查找）。
  - 上传后自动健康检查（向量条数、维度一致性、可检索性）。
- 检索前置校验
  - 在 similaritySearch 前校验 query vector 与表维度。
  - 统一错误码（如 `EMBED_DIM_MISMATCH`、`EMBED_PROVIDER_UNREACHABLE`）。

### 交付物
- 数据巡检脚本与后台任务。
- 一致性修复 CLI（可按 workspace 执行）。
- 检索维度诊断接口（供设置页与运维页展示）。

### 验收标准
- 任意上传失败不会留下脏数据。
- 维度不一致错误可被用户明确理解并给出处理建议。
- “上传后立即问答”命中成功率显著提升（定义基准样本集评估）。

### 风险与应对
- 风险：自动清理误删数据。  
  应对：先 Dry-run 输出，再人工确认；默认仅告警模式。

---

## Phase 2（M2）：多 Provider 能力层（LLM/Embedding）

### 目标
- 建立 Provider 抽象层，稳定支持 Ollama、GLM，并为后续 OpenAI/Gemini 等预留扩展点。

### 工作项
- 后端 Provider 抽象
  - 统一接口：`chat`, `embedding`, `healthCheck`, `listModels`。
  - provider-adapter 目录化（`providers/ollama`, `providers/glm`, ...）。
  - Provider 能力声明（支持 dimensions、流式、函数调用等）。
- 配置体系升级
  - Provider 级配置模型（按 provider 独立存储并支持切换）。
  - Embedding 维度策略：固定维度、自动探测、按模型映射。
  - 模型探测与连通性测试按钮（设置页）。
- 兼容与降级策略
  - Provider 不可用时可回退到备用 provider（可选）。
  - API 错误统一封装与脱敏日志。

### 交付物
- Provider SDK 层。
- 设置页“模型连通性测试/维度检测”模块。
- Provider 运行状态接口（用于运维页）。

### 验收标准
- Ollama/GLM 切换无需改代码，仅靠配置生效。
- Embedding 维度不再出现“隐式漂移”导致的线上失败。

---

## Phase 3（M3）：文档处理异步化（队列化）

### 目标
- 上传接口快速返回，文档解析与向量化后台执行，可重试、可追踪。

### 工作项
- 引入任务队列（可先基于 PostgreSQL 任务表，后续可接 Redis/BullMQ）。
- 新增任务状态机：`pending/running/success/failed/retrying/cancelled`。
- 拆分上传链路为独立步骤
  - 解析任务
  - 分块任务
  - 向量化任务
  - 入库任务
- 前端文档面板增加任务进度、失败原因、重试按钮。
- 增加幂等控制：同 file_hash 重复提交只执行一次有效任务。

### 交付物
- `document_jobs` 表及 worker 进程。
- 文档处理状态 API + 前端状态展示。

### 验收标准
- 大文件上传不阻塞用户请求。
- 失败任务可重试，且不会重复写脏数据。

---

## Phase 4（M4）：检索增强（Recall + Precision）

### 目标
- 让“问到文档但答不到”显著减少，提升召回率和答案可信度。

### 工作项
- 检索策略增强
  - 向量检索 + 关键词检索（PostgreSQL FTS）混合检索。
  - 文件名优先、标题优先、字段加权。
  - Query Rewrite（简写扩展、同义词、分词规范化）。
- 结果重排
  - 增加 reranker 接口（可先规则重排，再引入模型重排）。
  - 去重策略：按文档与段落层级去重。
- 可调策略面板
  - topN、阈值、混合检索比例、重排开关可配置。

### 交付物
- 混合检索引擎模块。
- 检索调参 UI 与“检索调试视图”（显示每条召回分数来源）。

### 验收标准
- 对既定问题集，命中率/可回答率提升（设量化指标）。
- 用户可在 UI 自助调参解决“命中不足”问题。

---

## Phase 5（M5）：对话能力增强

### 目标
- 提升多轮对话质量，支持线程化与上下文治理。

### 工作项
- 引入会话线程（Thread）模型，支持一个知识库多会话并行。
- 上下文策略
  - 历史消息窗口动态裁剪。
  - 关键事实提取与摘要缓存。
  - 低质量历史过滤（避免噪声累积）。
- 引用增强
  - 引用默认展开、片段定位、来源排序解释。
  - “回答-引用一致性检查”提示（可选）。

### 交付物
- 线程 API + 前端线程管理 UI。
- 上下文组装策略引擎。

### 验收标准
- 多轮问答连贯性提升，引用更稳定、可解释。

---

## Phase 6（M6）：权限与审计

### 目标
- 从“单管理员系统”升级到可协作、可审计的企业应用。

### 工作项
- RBAC：super admin / admin / editor / viewer。
- workspace 级权限控制（可见、可上传、可配置、可问答）。
- 审计日志
  - 关键操作：登录、配置变更、文档上传删除、权限变更。
  - 审计查询与导出。
- 安全策略
  - API Key 脱敏显示。
  - 可选 IP 白名单与登录风控策略。

### 交付物
- 权限模型与中间件升级。
- 审计日志表与审计页面。

### 验收标准
- 非授权用户无法访问越权资源。
- 关键操作可追溯到具体用户与时间。

---

## Phase 7（M7）：运维与可观测性

### 目标
- 具备线上稳定运行能力，故障可发现、可定位、可恢复。

### 工作项
- 指标监控
  - 请求延迟、错误率、上传耗时、向量化耗时、检索命中率。
  - Provider 可用性与调用失败率。
- 日志与追踪
  - request_id 全链路日志。
  - 错误分类与聚合告警（Webhook/邮件）。
- 备份恢复
  - DB 定时备份、恢复演练脚本。
  - 文档元数据与向量一致性快照。

### 交付物
- 运维面板（基础版本）。
- 备份恢复手册和演练记录。

### 验收标准
- 可在 30 分钟内定位主要线上故障原因（内部目标）。
- 具备可验证的数据恢复能力。

---

## Phase 8（M8）：测试体系与发布流程

### 目标
- 建立“能持续迭代而不破坏线上”的质量体系。

### 工作项
- 自动化测试
  - 后端单测：解析、分块、检索、权限。
  - 集成测试：上传→向量化→问答全链路。
  - 前端关键流程测试：设置、上传、问答、引用。
- 发布流程
  - lint + typecheck + test + build 作为强制门禁。
  - 分支策略：feature → staging → main。
  - 灰度发布与回滚脚本。

### 交付物
- CI 流水线配置与测试报告模板。
- 发布检查清单（Release Checklist）。

### 验收标准
- 每次发布前自动跑完关键回归。
- 故障可快速回滚到上一个稳定版本。

---

## Phase 9（M9）：生态扩展（MCP/连接器/插件）

### 目标
- 在不牺牲稳定性的前提下，引入扩展生态能力。

### 工作项
- MCP Server 接入基础框架（白名单 + 权限隔离）。
- 连接器扩展（先做 1~2 个高价值连接器）：
  - Git 仓库文档同步
  - Confluence（或企业内部知识源）
- 插件化接口规范
  - 输入输出契约
  - 错误码与超时控制

### 交付物
- 插件/连接器开发模板。
- 至少 2 个可用连接器。

### 验收标准
- 连接器可稳定增量同步，不影响主服务稳定性。

---

## Phase 10（M10）：产品化打磨与组织落地

### 目标
- 完成从“技术可用”到“团队可持续运营”的最后一公里。

### 工作项
- 管理能力
  - 配置版本化（变更记录、回滚）。
  - 知识库健康评分（覆盖率、命中率、新鲜度）。
- 性能优化
  - 热点缓存、分页优化、索引优化、批量写入优化。
- 组织落地
  - 运维手册、排障手册、值班流程。
  - 业务团队培训材料（使用规范、提示词规范、知识库维护规范）。

### 交付物
- v2 最终版本文档包（架构图、接口文档、运维手册、培训材料）。

### 验收标准
- 新成员可在 1 天内完成部署、配置、上传、问答全流程。
- 业务团队可独立维护知识库并稳定使用。

## 6. 成功标准（v2 完成定义）

- 上传成功后可检索性稳定，不再出现“有文档但答不到”。
- Provider 切换不需要改代码，配置可验证、可观测。
- 系统支持多人协作与权限隔离，关键操作可审计。
- 发布可回归、可灰度、可回滚。
- 内网环境可稳定运行并可持续演进。

## 7. 30 / 60 / 90 天执行计划（可直接落地）

> 说明：以下计划按“单产品负责人 + 1~2 后端 + 1 前端 + 0.5 测试/运维”配置设计。  
> 若资源更少，优先保证 30 天目标；若资源更充足，可并行推进 60/90 天任务。

### Day 1 - Day 30（稳定性与可用性冲刺）

#### 本阶段目标
- 用户上传后可稳定检索与问答。
- 模型/向量维度不再导致线上阻塞。
- 形成可定位问题的最小可观测体系。

#### 任务包 A：数据一致性治理（M1）
- A1. 建立 `documents-vectors` 一致性巡检脚本（CLI）
  - 输出：按 workspace 统计 `doc_chunks / vec_chunks / orphan_docs / orphan_vectors`。
  - 支持 `--dry-run` 与 `--fix` 模式。
- A2. 上链路补偿策略标准化
  - 上传链路所有中间步骤失败可回滚。
  - 对“同 file_hash 但无向量”的历史脏记录自动清理再重传。
- A3. 文件名编码标准化
  - 存储前 UTF-8 规范化，历史 latin1 兼容读取和删除。

#### 任务包 B：Provider 与 Embedding 稳定化（M2 部分）
- B1. Provider 配置模型标准化
  - 明确 provider、base_url、api_key、model、dimensions 的优先级（DB > ENV > default）。
- B2. Embedding 维度诊断
  - 增加“数据库维度 / 配置维度 / 实际返回维度”诊断 API。
- B3. 维度防护
  - 写向量前、检索前双重维度检查。
  - 对不稳定返回做归一化（padding/截断）并记录警告日志。

#### 任务包 C：检索兜底与可解释性（M4 部分）
- C1. 检索 fallback
  - 严格阈值失败后执行低阈值重试。
  - 文件名关键词匹配兜底（快速开始/安装说明类问题）。
- C2. 来源可视化
  - 引用来源默认展开，显示命中文档和片段。
- C3. 调试日志
  - 输出 topN 原始距离、过滤阈值、最终命中数量。

#### 任务包 D：回归与上线门禁（M8 部分）
- D1. 最小回归脚本
  - 场景：上传 3 文档→问 3 类问题→验证命中来源。
- D2. 发布门禁（最小版）
  - 必过：`build` + 核心链路脚本 + 健康检查。

#### Day 30 验收标准
- “上传成功但无法命中”问题显著下降（以内部问题集统计）。
- 所有维度错误转为可读错误，不再出现数据库底层异常直出。
- 新增 1 套可重复运行的回归脚本。

#### Day 30 输出物
- 巡检/修复 CLI
- 维度诊断 API
- 检索 fallback 与来源可视化
- 最小发布门禁脚本

---

### Day 31 - Day 60（能力增强与工程化）

#### 本阶段目标
- 文档处理异步化，支持失败重试与状态可见。
- 检索从“能用”升级为“可调优、可解释、召回更稳”。
- 具备基础监控与告警。

#### 任务包 E：异步化文档处理（M3）
- E1. `document_jobs` 任务表与 worker
  - 状态机：`pending/running/success/failed/retrying/cancelled`。
  - 记录每步耗时、失败原因、重试次数。
- E2. 上传接口改造
  - 上传请求快速返回 job_id。
  - 前端轮询/订阅 job 状态并展示进度条。
- E3. 幂等与并发控制
  - 同文件重复提交只保留一个有效任务。
  - 限制单 workspace 同时处理任务数。

#### 任务包 F：检索增强（M4）
- F1. 混合检索（向量 + FTS）
  - 支持权重融合（如 `0.7 * vector + 0.3 * keyword`）。
- F2. 结果重排（规则版）
  - 文件名命中加分、标题命中加分、重复段落降权。
- F3. 调参界面
  - 暴露 `topN / threshold / hybrid_weight / filename_boost`。
  - 增加“检索调试视图”查看每条分数构成。

#### 任务包 G：可观测与运维基础（M7 部分）
- G1. 指标采集
  - 上传成功率、向量化失败率、检索命中率、平均响应耗时。
- G2. 告警与日志
  - 维度不一致、Provider 不可达、任务重试超限触发告警。
- G3. 备份脚本（基础版）
  - DB 日备份与恢复演练文档。

#### Day 60 验收标准
- 上传链路异步化上线，失败可重试。
- 混合检索对“文件名明确问题”命中率明显提升。
- 运维可通过面板/日志快速定位故障点。

#### Day 60 输出物
- 异步任务系统（API + UI）
- 混合检索与调参面板
- 基础监控与告警规则

---

### Day 61 - Day 90（平台化能力与组织落地）

#### 本阶段目标
- 支持多人协作与权限隔离。
- 建立发布质量体系与回滚机制。
- 输出可复制的部署/运维/培训标准。

#### 任务包 H：权限与审计（M6）
- H1. RBAC 落地
  - 角色：super admin / admin / editor / viewer。
  - workspace 级授权（查看、上传、配置、问答）。
- H2. 审计日志
  - 覆盖登录、权限变更、配置变更、文档管理操作。
- H3. 审计页面与导出
  - 支持按用户、workspace、时间过滤。

#### 任务包 I：测试与发布体系（M8）
- I1. 自动化测试完善
  - 后端单测 + 集成测试 + 前端关键流程测试。
- I2. CI/CD 门禁
  - lint/typecheck/test/build 全通过才可发布。
- I3. 灰度与回滚
  - 版本标识、回滚脚本、回滚演练记录。

#### 任务包 J：产品化与团队可运营（M10）
- J1. 运维手册与排障手册
  - 常见 20+ 故障场景标准处理流程。
- J2. 业务培训材料
  - 知识库维护规范、提示词编写规范、命中排障流程。
- J3. 健康评分
  - workspace 维度展示：文档覆盖率、命中率、失败率、更新时间。

#### Day 90 验收标准
- 多角色协作流程跑通，越权访问被正确拦截。
- 发布流程稳定可复用，回滚可在短时间完成。
- 新成员可在 1 天内完成部署与基础运维。

#### Day 90 输出物
- RBAC + 审计系统
- CI/CD 与灰度回滚体系
- 完整运维/培训文档包

## 8. 每周执行节奏建议（适用于 30/60/90 全周期）

- 周一：确定本周目标与验收指标（只定 1~2 个主目标）。
- 周二~周四：开发 + 联调 + 日内小回归。
- 周五：集中回归、风险复盘、下周预备任务。
- 每周固定产出：
  - 本周变更清单
  - 风险清单与处理结果
  - 验收记录（通过/失败原因/下一步）

## 9. 风险总表（30/60/90 重点关注）

- 模型返回维度不稳定导致检索失败  
  - 策略：维度探测、归一化、检索前校验、可读错误。
- 异步队列引入后复杂度上升  
  - 策略：先 DB 队列再 Redis，保持可回滚。
- 检索增强后响应时延上升  
  - 策略：分层开关、缓存、默认轻量策略。
- 权限改造影响现有用户流程  
  - 策略：灰度角色策略 + 审计观察期。
- 发布频率提升导致质量波动  
  - 策略：CI 门禁 + 回滚演练常态化。
